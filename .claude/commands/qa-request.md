# QA Request Message

## Arguments
사용자 입력: $ARGUMENTS

## Context
- QA/검수 요청, 서비스 전달, 핸드오프 메시지를 고객 카톡에 복붙할 한국어 메시지로 생성
- /client-chat과 다름: client-chat = 자유 입력 → 변환. qa-request = 구조화된 전달물 → 정형 메시지.
- 노션 저장 없음 — 터미널에 바로 출력

## Delivery Type Detection
입력 내용을 분석하여 전달 유형을 자동 판단:

| Type | 감지 신호 |
|------|----------|
| **web** | 링크, URL, 서비스 |
| **web+admin** | 어드민, admin, 관리자, ID/PW |
| **apk** | apk, 안드로이드, android |
| **program** | zip, exe, 프로그램, 파일 |
| **testflight** | testflight, iOS, 초대 |
| **mixed** | 위 조합 |

애매하면 PM에게 확인 요청.

## Output Structure

```
안녕하세요, {context — 1줄: 내부 검토 완료 / 1차 개발 완료 / 수정사항 반영 등}

{전달물 블록 — 유형별 자동 구성}

[검수 페이지]
- {노션 링크}
- 피드백은 위 페이지에 남겨주시면 확인 후 반영하겠습니다.

{참고사항 — 있을 때만}

확인 부탁드리겠습니다. 감사합니다.
```

### 전달물 블록 — 유형별

**web:**
```
[서비스]
- 링크: {url}
- {테스트 안내}
```

**web+admin:**
```
[서비스]
- 링크: {url}
- {테스트 안내}

[어드민]
- 링크: {url}
- ID: {id}
- PW: {pw}
```

**apk:**
```
[APK 파일]
- {파일 전달 방식}
- 설치: 다운로드 → "알 수 없는 앱 설치 허용" → 설치 → 실행
- {테스트 안내}
```

**program:**
```
[프로그램 파일]
- {파일 전달 방식}
- 실행: 압축 해제 → {파일명} 더블클릭
- {테스트 안내}
```

**testflight:**
```
[TestFlight]
- {이메일}로 초대를 발송했습니다
- 메일함에서 "TestFlight" 초대 메일 확인 → TestFlight 앱 설치 → 앱 다운로드
- {테스트 안내}
```

## Instructions

1. **인자 파싱**: 클라이언트명과 전달물 정보 추출
2. **컨텍스트 로드**:
   - `clients/{client-name}/CLAUDE.md` — 클라이언트 톤, 기술 수준
   - `glossary/{client-name}.md` — 용어 일관성
3. **전달 유형 감지**: 입력에서 전달물 종류 판단
4. **메시지 작성**:
   - context 1줄: 왜 이걸 전달하는지 (내부 검토 완료, 1차 개발 완료, 수정 반영 등)
   - 전달물 블록: 유형별 템플릿에 맞춰 구성
   - 검수 페이지: 노션 링크 + 피드백 방법 안내
   - 참고사항: 있을 때만 (테스트 시 주의사항, 특정 기능 안내 등)
5. **톤**: 클라이언트 CLAUDE.md 톤 우선 적용. 없으면 합니다체.
   - 기술 용어는 클라이언트 수준에 맞게 변환
   - 설치/실행 안내는 비기술자도 따라할 수 있게
6. **한국어로 작성**: 카톡에 바로 복붙
7. **출력만**: 노션 저장 없음

## Rules
- 전달물 정보가 부족하면 PM에게 확인 요청 (링크, 계정, 파일 등)
- 검수 페이지 링크가 없으면 "[검수 페이지 링크 추가 필요]"로 placeholder
- 클라이언트가 개발 모르는 경우 설치/실행 안내를 최대한 쉽게
- 각 블록은 [] 대괄호 헤더로 구분 — 카톡에서 스캔하기 쉽게
- 불필요한 인사말/설명 최소화 — 전달물 중심
- 여러 전달물이 있으면 블록을 순서대로 나열 (mixed)
